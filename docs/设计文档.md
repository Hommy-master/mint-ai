# Mint-AI REST API 设计说明

## 1. 概述

本设计面向「**输入视频、一键成片**」的后端服务：用户提交原始视频的访问地址，系统在后台按固定 **三步流水线** 完成初切、AI 高光与清洗、以及基于 **capcut-mate** 的剪辑与成片导出。对外采用 **FastAPI** 提供 RESTful API，运行时自动生成 **OpenAPI 文档**（Swagger UI / ReDoc），便于联调与契约对齐。

| 目标 | 说明 |
|------|------|
| 文档 | 服务启动后可通过浏览器访问交互式 API 文档 |
| 版本 | URL 路径级版本号，避免破坏性变更影响旧客户端 |
| 参数 | 使用 Pydantic 模型做 **严格类型与约束校验**（请求体、查询参数） |
| 并发 | 路由处理函数采用 **async**；长耗时步骤（下载、FFmpeg、调用大模型与 capcut-mate）宜放在 **任务队列 / Worker** 中执行，避免阻塞 API 事件循环 |

**包管理**：工程统一使用 **[uv](https://github.com/astral-sh/uv)** 管理 Python 版本、虚拟环境与依赖（`pyproject.toml` + 锁文件）。

---

## 2. 成片业务流程（三步）

以下为用户提交任务后，服务端应遵循的 **业务主流程**。各步产出可为中间文件或内存中的任务状态，具体存储策略在实现阶段确定。

### 第一步：初步切片

| 项目 | 说明 |
|------|------|
| **目的** | 将输入长视频按时间维度切分为便于后续分析的单元 |
| **产出** | 时长约 **20 分钟** 左右的视频片段（若原片不足 20 分钟，则按实际时长一段或多段处理，规则可在实现时细化） |
| **实现要点** | 通常基于 **FFmpeg** 按时间轴切片；需处理下载源视频、容器格式与编码兼容性 |

### 第二步：AI 分析、素材清洗与高光切片

| 项目 | 说明 |
|------|------|
| **目的** | 理解视频内容，去掉不适合成片的片段，保留可用于成片的高光时刻 |
| **处理** | 使用 **AI 大模型** 对第一步得到的片段（或其抽帧/音频特征）进行分析，完成 **素材清洗** 与 **高光片段** 的判定（时间范围或镜头列表） |
| **产出** | 经筛选的、可用于成片的高光片段；通过 **FFmpeg** 按判定的时间轴 **再次切片**，得到若干 **高光视频文件**（或等价的可寻址媒体片段），供第三步剪辑使用 |

### 第三步：基于 capcut-mate 的剪辑与成片

| 项目 | 说明 |
|------|------|
| **依赖** | 开源项目 **capcut-mate**（通过其提供的接口接入） |
| **能力** | 在第二步的高光片段基础上，完成 **视频剪辑**，并包含但不限于：**转场**、**字幕**、**花字**、**BGM** 等处理（以 capcut-mate 实际暴露的能力为准） |
| **成片** | 调用 capcut-mate 提供的 **`gen_video`**（或文档约定的等价接口）**生成最终视频** |
| **实现要点** | 需部署或与 capcut-mate 服务联通；参数映射、异步回调或轮询与任务查询接口对齐 |

### 流程关系小结

```text
输入 video_url
    → [1] 初步切片（≈20 分钟/段）
    → [2] 大模型分析 + 清洗 + 高光判定 → FFmpeg 切出高光片段
    → [3] capcut-mate：剪辑（转场 / 字幕 / 花字 / BGM 等）→ gen_video → 成片
```

## 3. API 版本与基础路径

- **策略**：在路径中包含主版本号，例如 `v1`。主版本升级表示不兼容的 URL、字段或语义变更。
- **基础前缀**：`/openapi/v1`（所有对外业务接口均挂在该前缀下）

后续若发布 `v2`，可并行暴露 `/openapi/v2/...`，`v1` 在约定期内继续维护。

> **命名说明**：业务路由以 `/openapi/...` 开头，与 FastAPI 用于暴露 **机器可读 OpenAPI 规范文件** 的默认路径 `/openapi.json` 不同；后者为框架元数据，前者为业务 API 命名空间。若团队希望进一步减少歧义，实现时可将 `openapi_url` 自定义为例如 `/schema/openapi.json`（见 §5.3）。

## 4. 统一响应结构

所有业务接口在 HTTP 200 且业务成功时，响应体采用统一外层结构（便于前端与监控解析）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | `integer` | 业务码，`0` 表示成功；非 `0` 表示失败（与 HTTP 状态码配合使用时可再约定） |
| `message` | `string` | 人类可读说明 |
| `data` | `object \| null` | 业务数据；无数据时为 `null` 或空对象（见各接口） |

> **说明**：具体「失败时 HTTP 状态码取 4xx/5xx 还是仍返回 200 + code」可在实现阶段固定一条团队约定；本设计文档以接口字段与语义为主。

## 5. 接口设计

### 5.1 提交成片任务

客户端仅提交 **输入视频地址**；服务端创建异步任务，按 **§2 三步流程** 在后台执行。

| 项目 | 说明 |
|------|------|
| **方法** | `POST` |
| **URL** | `/openapi/v1/tasks` |
| **Content-Type** | `application/json` |

**请求体（JSON）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `video_url` | `string` | 是 | 待处理原始视频的访问地址（建议约束为 `http`/`https`，长度上限在实现时用 Pydantic 校验） |

**请求示例**

```json
{
  "video_url": "https://example.com/video.mp4"
}
```

**成功响应示例**（HTTP 200）

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "task_id": "xxx"
  }
}
```

**`data` 字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `task_id` | `string` | 任务唯一标识，用于后续查询 |

---

### 5.2 查询任务状态 / 结果

用于轮询任务进度；`data` 中宜体现当前处于 **第几步** 以及是否已产出 **成片地址**（实现时填充）。

| 项目 | 说明 |
|------|------|
| **方法** | `GET` |
| **URL** | `/openapi/v1/tasks` |
| **Query** | `task_id`（必填） |

**请求示例**

```http
GET /openapi/v1/tasks?task_id=xxx
```

**Query 参数**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `task_id` | `string` | 是 | 提交任务时返回的 `task_id` |

**成功响应示例**（HTTP 200）

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

> **说明**：`data` 建议逐步扩展，例如：`status`（pending / running / succeeded / failed）、`step`（1～3 与 §2 对应）、`progress`、`error_message`、**成片 `result_url`**（第三步 `gen_video` 完成后）等；字段名与枚举在实现阶段与 OpenAPI schema 对齐即可。

---

### 5.3 文档与健康检查（建议）

| 项目 | 说明 |
|------|------|
| OpenAPI 规范 JSON | FastAPI 默认 `/openapi.json`（实现时可改为自定义路径，避免与业务前缀 `/openapi/v1` 在运维侧混淆） |
| Swagger UI | 默认 `/docs` |
| ReDoc | 默认 `/redoc` |
| 健康检查（可选） | `GET /health`，用于负载均衡探活，返回简单 JSON 即可 |

以上路径若与网关前缀拼接，需在部署文档中说明对外暴露的完整 URL。

## 6. 严格参数校验（实现约定）

1. **请求体**：使用 Pydantic `BaseModel`，对 `video_url` 做类型、`min_length`、URL scheme（如仅允许 `http`/`https`）等约束。
2. **查询参数**：`task_id` 使用 Pydantic 模型或 FastAPI `Query(...)`，禁止空字符串（若业务不允许）。
3. **校验失败**：返回 **422**，响应体为 FastAPI 默认验证错误结构；若产品要求与「统一 `code`/`message`」完全一致，可在后续迭代中增加全局异常处理器（设计层面预留即可）。

---

## 7. 异步与运行时（实现约定）

- **API 层**：路由使用 `async def`，仅负责任务入队与查询；**不**在请求线程内长时间执行 FFmpeg 或阻塞式调用 capcut-mate。
- **Worker**：第一步、第二步中的 FFmpeg 与 **大模型调用**，以及 **第三步 capcut-mate 剪辑与 `gen_video`**，均宜在独立工作进程或任务队列中执行，并更新任务状态供 **§5.2** 查询。
- **外部依赖**：运行环境需可用的 **FFmpeg**；**capcut-mate** 需按该项目文档完成部署与网络连通；**大模型** 的 API Key、限流与超时在 `core/config` 中配置。

## 8. 工程目录结构（建议）

在 **§2 三步流程** 下，源码建议按「流水线阶段」与「基础设施」拆分；以下为基于 FastAPI + uv 的推荐布局：**应用入口脚本放在仓库根目录的 `main.py`**。

```text
mint-ai/
├── main.py                 # 入口：启动 Uvicorn（支持 uv run main.py）
├── pyproject.toml
├── uv.lock
├── README.md
├── docs/
│   └── 设计文档.md
├── src/
│   ├── app.py              # FastAPI 实例、挂载路由、CORS、生命周期
│   ├── api/
│   │   └── v1/
│   │       ├── router.py     # 聚合 v1（前缀 /openapi/v1）
│   │       └── tasks.py      # 提交 / 查询任务
│   ├── schemas/
│   │   ├── common.py         # 统一响应体、业务码等
│   │   └── tasks.py          # Task 相关 Pydantic 模型
│   ├── services/
│   │   ├── task_service.py   # 任务创建、状态持久化、查询
│   │   ├── pipeline/         # 三步流水线（可按子模块再拆）
│   │   │   ├── step1_initial_slice.py   # 第一步：初步切片（≈20 分钟）
│   │   │   ├── step2_ai_highlights.py     # 第二步：大模型 + 清洗 + FFmpeg 高光
│   │   │   └── step3_capcut_mate.py       # 第三步：capcut-mate 剪辑 + gen_video
│   │   ├── ffmpeg_ops.py     # FFmpeg 封装（若多处复用）
│   │   └── capcut_mate_client.py  # capcut-mate HTTP/CLI 客户端封装（可选）
│   └── core/
│       └── config.py         # 环境变量：模型、capcut-mate 地址、FFmpeg 路径等
└── tests/
    └── api/
        └── test_tasks.py
```

**分层说明（简要）**

| 目录/文件 | 职责 |
|-----------|------|
| `main.py`（根目录） | 进程入口：`uv run main.py`；加载 `src/app.py` 中 `app`（需将 `src` 加入搜索路径或使用 `--app-dir src`） |
| `src/app.py` | 组装 FastAPI 应用 |
| `src/api/v1/` | HTTP 层：路由与参数校验 |
| `src/schemas/` | Pydantic 模型 |
| `src/services/pipeline/` | 与 §2 三步一一对应；内部编排 FFmpeg、大模型与 capcut-mate |
| `src/core/` | 配置与共享工具 |

## 9. uv 使用说明（工程约定）

- **初始化项目**：在仓库根目录使用 `uv init` 或手写 `pyproject.toml`，依赖通过 `uv add ...` 添加；**src 布局**下在 `pyproject.toml` 中配置包发现/可编辑安装指向 `src/`。
- **运行（推荐）**：**`uv run main.py`**。
- **等价方式（可选）**：`uv run uvicorn app:app --app-dir src --reload`。
- **锁文件**：将 `uv.lock` 纳入版本控制。
