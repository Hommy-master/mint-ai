# Mint AI Backend 设计文档

## 1. 背景与目标

基于当前 `backend` 代码（已保留 `user/upload/wxgate` 三大模块），本设计文档定义以下目标能力：

1. 微信扫码登录，登录成功后签发 JWT Token；
2. 手机号 + 密码登录，登录成功后签发 JWT Token；
3. VIP 支持 3 个等级（1/2/3），不同等级权益不同，且有效期不同；
4. 同时支持微信扫码支付 + 支付宝扫码支付；用户可查看消费记录（含 VIP 购买记录 + 积分消费记录）；
5. 支持分销：分享链接 `?from=userid`，被邀请用户消费后邀请人获 10% 提成；支持提成提现；支持查看下级消费总额与分销收益；
6. 支持积分单独购买。


## 2. 当前代码基线（摘要）

当前已具备/部分具备能力：

- 登录：
  - 手机号+密码登录：已实现；
  - 微信扫码登录：已实现（`wxgate` 回调 + ticket 轮询）；
  - JWT：已实现签发与认证中间件；
- 支付：
  - 微信支付 Native：已实现（下单、回调、查单）；
  - 支付宝：未实现；
- 积分/VIP：
  - 已有 `users.vip_level`、`users.vip_expire_at`；
  - 已有充值、消费日志；
  - VIP 当前仅包含等级与有效期权益，不包含自动赠分；
- 分销：
  - `users.referrer_id` 字段已存在；
  - 邀请收益、提现、统计尚未落地。


## 3. 总体架构设计

采用分层结构：

- `api`：请求/响应结构定义；
- `internal/controller`：参数校验、鉴权、调用 service；
- `internal/service`：业务编排（登录、支付、分销、积分、VIP）；
- `internal/pkg/db`：数据库访问与事务封装；
- `internal/model`：GORM 模型；
- `internal/middleware`：JWT 鉴权、统一响应；
- `internal/cmd`：服务启动与环境初始化（建表）。

关键原则：

- 支付成功回调做幂等（`order_no` 唯一约束 + 状态机）；
- 涉及“金额、积分、分佣、提现”的更新全部走事务；
- 金额统一用“分”（整数）存储，避免浮点精度问题；
- 重要业务表增加审计字段与状态字段，支持后续运营对账。


## 4. 功能设计

### 4.1 微信扫码登录 + JWT

- 流程：
  1. 前端请求二维码 ticket；
  2. 用户微信扫码，微信侧回调 `wxgate`；
  3. 服务端将 `ticket -> openid` 写入缓存；
  4. 前端轮询 ticket 状态；
  5. 命中成功后查询/创建用户，签发 JWT。

- 关键点：
  - ticket 有效期控制（如 300s）；
  - 同一 ticket 仅允许一次成功登录；
  - OpenID 与用户唯一关联。


### 4.2 手机号 + 密码登录 + JWT

- 流程：
  1. 输入手机号、密码；
  2. 校验用户，若首次登录可按现有策略自动注册；
  3. 签发 JWT（含 `userID`、`exp`）。

- 安全建议：
  - 密码从 MD5 升级到 `bcrypt/argon2`（建议演进项）；
  - 登录限流与失败次数风控。


### 4.3 VIP（3等级 + 有效期）

- 会员等级定义：
  - VIP1：等级 1；
  - VIP2：等级 2；
  - VIP3：等级 3。

- 会员权益机制：
  - 通过 `vip_plans` 配置各等级价格、有效期、折扣策略与权益描述；
  - 支付成功后升级对应等级并刷新有效期；
  - 过期后按普通用户权益处理。

- 有效期：
  - VIP 购买后按方案叠加有效期；
  - 续费时在当前有效期基础上顺延；
  - 降级/过期由任务或查询时实时计算。


### 4.4 双支付（微信 + 支付宝）与消费记录

- 支付通道：
  - `WECHAT`：沿用现有微信 Native；
  - `ALIPAY`：新增统一下单接口与回调处理；
  - 统一订单表 `orders` 记录通道、状态、三方交易号。

- 消费记录：
  - `orders`：记录支付订单（含 VIP 购买、积分购买）；
- `points_ledger`：记录积分变动（充值、消费、扣减、退款）；
  - 页面聚合：`支付记录 + 积分明细 + VIP购买记录`。


### 4.5 分销（10%提成）+ 提现 + 统计

- 邀请关系：
  - 分享链接 `?from=userid`；
  - 注册/首登时写入 `users.referrer_id`（仅首次绑定，不可更改）。

- 分佣规则：
  - 被邀请用户“支付成功金额”按 10% 计算分佣；
  - 分佣落账到 `commission_ledger`；
  - 钱包余额放在 `user_wallets.commission_balance`。

- 提现：
  - 用户发起提现申请 `withdraw_requests`；
  - 后台审核/打款后更新状态（`PENDING/APPROVED/PAID/REJECTED`）；
  - 提现成功时扣减可提现余额并记流水。

- 统计：
  - 下级总消费：按 `referrer_id` 关联下级订单聚合；
  - 总收益：按分佣流水聚合；
  - 已提现/待提现分开展示。


### 4.6 积分单独购买

- 产品类型新增（或沿用）`POINTS_PACKAGE`；
- 下单后支付成功直接增加积分余额；
- 积分发放写入积分流水；
- 支持固定档位与自定义金额。

## 6. 数据库表设计

> 设计目标：**最小化实现 + 可扩展**。  
> 原则：优先复用 `users / orders / plugin_balance_logs`，通过少量字段扩展实现 VIP、分销、提现。

MVP 最少表清单（5张）：

1. `users`
2. `orders`
3. `plugin_balance_logs`
4. `withdraw_requests`
5. `uploads`

### 6.1 users（核心用户表，保留并小幅扩展）

字段建议：

- `id` bigint PK
- `name` varchar(64) unique
- `phone` varchar(64) unique
- `pass` varchar(128)
- `wechat_id` varchar(64) unique
- `email` varchar(256)
- `points` bigint                            # 当前积分（整数）
- `vip_level` int                            # 0/1/2/3
- `vip_expire_at` datetime
- `referrer_id` bigint default 0 index       # 邀请人用户ID
- `commission_balance_cent` bigint default 0 # 当前可提现分佣余额（分）
- `commission_total_cent` bigint default 0   # 累计分佣（分）
- `commission_withdrawn_cent` bigint default 0 # 累计已提现（分）
- `ext` varchar(1024)                        # 扩展字段(JSON)
- `created_at` / `updated_at` / `deleted_at`

索引：

- `uk_users_phone`
- `uk_users_wechat_id`
- `idx_users_referrer_id`
- `idx_users_vip_level_expire`


### 6.2 orders（核心订单表，统一支付与分销快照）

用途：统一支付订单（微信/支付宝、VIP购买、积分购买）。

核心字段：

- `id` bigint PK
- `user_id` bigint index
- `order_no` varchar(64) unique
- `product_type` varchar(32)       # VIP / POINTS / OTHER
- `product_id` varchar(64)
- `pay_method` varchar(32)         # WECHAT / ALIPAY
- `amount_cent` bigint             # 订单金额（分）
- `order_status` varchar(32)       # CREATED/PAYING/PAID/CLOSED/REFUNDED
- `transaction_id` varchar(64)
- `payed_at` datetime
- `from_user_id` bigint default 0  # 邀请来源（下单快照）
- `commission_amount_cent` bigint default 0 # 本单产生分佣金额（分）
- `commission_status` varchar(16) default 'NONE' # NONE/PENDING/SETTLED
- `ext` varchar(1024)              # 扩展字段(JSON)
- `created_at` / `updated_at` / `deleted_at`

索引：

- `uk_orders_order_no`
- `idx_orders_user_id_created_at`
- `idx_orders_from_user_id`
- `idx_orders_status`


### 6.3 plugin_balance_logs（积分流水，复用现有表）

用途：统一记录积分增减明细。

字段：

- `id` bigint PK
- `user_id` bigint index
- `points` bigint                  # 正负值，正为增加，负为扣减
- `order_no` varchar(64) index
- `description` varchar(128)       # RECHARGE / CONSUME / REFUND / VIP / MANUAL 等
- `ext` varchar(1024)              # 扩展字段(JSON，可存balance_after等)
- `created_at` / `updated_at` / `deleted_at`

说明：

- MVP 不新增 `points_ledger` 表，直接复用现有 `plugin_balance_logs`；
- 如后续流水查询与财务审计复杂度提升，再拆分专用账本表。


### 6.4 withdraw_requests（唯一建议新增表）

用途：提现申请与审核。

字段（最小化）：

- `id` bigint PK
- `user_id` bigint index
- `request_no` varchar(64) unique
- `amount_cent` bigint
- `status` varchar(16)                 # PENDING/APPROVED/PAID/REJECTED
- `channel` varchar(32)                # WECHAT / ALIPAY / BANK
- `account_no` varchar(128)
- `account_name` varchar(64)
- `audit_remark` varchar(256)
- `paid_at` datetime
- `order_no` varchar(64) index
- `created_at` / `updated_at` / `deleted_at`

> 注：`order_no` 可用于关联提现打款单号（若无可为空）。


### 6.5 uploads（保留）

保持现有结构，作为上传业务记录，与本次方案无冲突。

### 6.6 本期不新增但预留扩展

- `vip_plans`：当前可用配置文件/常量维护 VIP 档位；后续若需运营后台可落库；
- `referral_relations`：当前通过 `users.referrer_id` 即可满足绑定关系；
- `commission_ledger`：当前可通过 `orders.commission_*` + `users` 钱包字段支撑；
- 当对账、审计、多渠道分润复杂化时，再拆分独立流水表。


## 7. 关键业务流程（简版）

### 7.1 支付成功统一处理

1. 校验签名/回调参数；
2. 幂等判断订单状态；
3. 订单置为 `PAID`；
4. 按产品类型执行业务：
   - VIP：升级等级、顺延有效期；
   - 积分包：增加积分；
5. 写积分流水；
6. 若存在邀请关系，计算 10% 分佣，更新 `orders.commission_*` 与 `users` 钱包字段；
7. 事务提交。


## 8. 接口设计建议（增量）

- 登录：
  - `POST /openapi/v1/user/auth/login`
  - `GET /openapi/v1/user/auth/qrcode`
  - `GET /openapi/v1/user/auth/qrcode/status`
- 支付：
  - `POST /openapi/v1/payment/create-order`
  - `POST /openapi/v1/payment/wechat-notify`
  - `POST /openapi/v1/payment/alipay-notify`
  - `GET /openapi/v1/payment/order-status`
- 记录：
  - `GET /openapi/v1/user/orders`
  - `GET /openapi/v1/user/points/logs`
- 分销：
  - `GET /openapi/v1/user/referral/summary`
  - `GET /openapi/v1/user/referral/orders`
  - `POST /openapi/v1/user/withdraw/apply`
  - `GET /openapi/v1/user/withdraw/logs`


## 9. 迁移与落地计划

1. **第一阶段（兼容）**  
   在现有表上做最小增量改造：扩展 `users`、`orders` 字段，并新增 `withdraw_requests`。

2. **第二阶段（支付统一）**  
   抽象 `payment core`，接入支付宝扫码支付，统一订单状态流转。

3. **第三阶段（分销/提现）**  
   上线邀请绑定、分佣入账、提现申请与审核流程。

4. **第四阶段（数据治理）**  
   补充报表、对账与监控告警。


## 10. 非功能要求

- 安全：支付回调签名校验、JWT 过期与刷新策略、接口限流；
- 一致性：订单、积分、分佣、钱包更新必须事务化；
- 可观测：关键链路日志（下单、回调、分佣、提现）；
- 幂等性：回调重复投递不重复入账。
